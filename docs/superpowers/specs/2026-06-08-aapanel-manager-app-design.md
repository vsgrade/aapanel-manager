# Дизайн: Web-приложение управления aaPanel (многосерверная панель)

> Спецификация (источник истины для плана реализации). Дата: 2026-06-08.
> Фаза проекта: переход от документации API к **приложению** (см. [CLAUDE.md](../../../CLAUDE.md), [PROJECT_RULES.md](../../../PROJECT_RULES.md) §9–11).
> Статус: согласовано с пользователем в брейншторме. Далее → план реализации (skill `writing-plans`).

## 1. Цель и контекст

Веб-приложение для управления **многими серверами aaPanel** через **бэкенд-прокси**: браузер с панелями напрямую не общается, секреты (`api_sk`) живут только на сервере. Опирается на снятую и проверенную документацию API (`docs/{en,ru}/*`) и обёртку `examples/javascript/aapanel-client.ts`.

**Главная находка проекта** (используем): один постоянный `api_sk` на корне панели покрывает и `/system?action=…`, и внутренние `/v2/...` — приложению хватает одного стабильного ключа на сервер.

**Цель продукта:** компактный, красивый, быстрый дашборд; привлекательный для сообщества (звёзды на GitHub).

## 2. Объём (scope)

### В составе MVP
- **Серверо-центричная навигация:** Уровень 1 — таблица серверов; Уровень 2 — полное управление выбранным сервером.
- **Управление серверами** (роль admin): добавить/изменить/удалить сервер (URL + `api_sk`, шифрование в покое).
- **Аутентификация приложения:** несколько пользователей + роли **admin** / **viewer** (viewer = только чтение).
- **Обзор сервера:** живые метрики (CPU/RAM/диск).
- **Проекты сервера:** единая таблица **всех типов** (PHP, Node, Proxy, Python, Go) с колонкой «Тип» и фильтром; действия (старт/стоп/рестарт/логи/инфо) — по возможностям типа. Node — первым (API снят глубже всего), остальные типы — по мере снятия их API.
- **Кеш статусов + фоновый воркер-опрос + SSE** (живой дашборд).
- **Двуязычие** RU + EN (next-intl).
- **Аудит действий** (кто/что/когда/результат).

### Итеративно после MVP (заложить место в архитектуре)
- Разделы сервера: Базы данных, Файлы, FTP, Cron, Firewall (UI поверх уже снятого API).
- UI просмотра аудита.

### Вне scope (сейчас)
- Облачный раздел «Домены» aaPanel (требует внешнего аккаунта).
- Живой терминал/логи по WebSocket (будущее; тогда добавим Socket.IO точечно).
- Масштаб «тысячи серверов» (закладываем границы, реализуем при необходимости).
- Запись правил firewall/WAF/SSH/SSL (security-sensitive; добавляем под отдельные требования; API снимать рецептом).

## 3. Масштаб и нефункциональные требования

- **Целевой масштаб:** сотни серверов (~100–500). Архитектура «design for scale» к тысячам без переписывания.
- Production-ready, без костылей/заглушек; полная типизация (строгий TS); продуманная обработка ошибок и логирование; оптимизация загрузки (code splitting, dynamic import, кеширование, виртуализация таблиц).
- Кроссплатформенность: разработка на Windows, прод на Ubuntu (без OS-специфичных путей, LF).

## 4. Архитектура (подход B: фоновый сбор + кеш + живые действия)

- **Один Next.js (App Router)** + **воркер** (опрос) + **PostgreSQL**.
- **Чтение (быстро):** Server Components читают **кеш статусов** из Postgres → списки открываются мгновенно даже на сотнях серверов.
- **Детали и все действия (свежо):** Server Actions ходят **вживую** на нужную панель через типизированный клиент (на базе `aapanel-client.ts`, режим `api_sk`).
- **Воркер:** цикл (node-cron) с **ограниченной параллельностью** опрашивает серверы, апсертит статус в Postgres; корректная обработка оффлайн/таймаутов; ретраи с backoff для временных ошибок.
- **Real-time (браузер):** **SSE** (сервер→браузер) для живого дашборда. **Адаптивная частота:** глобальная таблица ~60 сек; открытый конкретный сервер ~3–5 сек (опрашивается только он).
- **Почему не Socket.IO к панелям:** у aaPanel нет push/WebSocket-API для статистики (её UI сам опрашивает HTTP — `GetSystemTotal`/`GetNetWork`). Сбор с источника — только опросом по HTTP. Socket.IO зарезервирован на будущее (двусторонние фичи, напр. живой терминал aaPanel по WS).

## 5. Стек

- **Next.js (App Router) + TypeScript (strict)** — RSC + Server Actions, бэкенд-прокси.
- **PostgreSQL + Prisma** — пользователи/роли, серверы (шифр. креды), кеш статусов, аудит.
- **Auth.js (NextAuth v5)** — credentials, сессии, роли admin/viewer.
- **UI:** Tailwind CSS + **shadcn/ui** (стиль **base-nova / Base UI**, не Radix — триггеры через `render={…}`) + **Framer Motion** + **TanStack Table v8** (плотные таблицы; v9-alpha не используем). **TanStack Query** отложен до Фазы 3 (нужен для живых SSE-обновлений).
- **Валидация:** zod; формы — **Server Actions + `useActionState`/`useTransition`** (react-hook-form не используется — меньше клиентского кода, чище для RSC).
- **Самоподписанный TLS:** `undici` (явная зависимость) — `Agent` с `rejectUnauthorized:false` как `dispatcher` глобального fetch; создаётся лениво и только при `insecureTLS`.
- **Логирование:** pino (структурное, серверное).
- **i18n:** next-intl (RU + EN).
- **Тесты:** Vitest (unit/интеграция) + Playwright (e2e).
- **Рантайм:** **Node 24 LTS** (точная минорная версия — сверить с поддерживаемым диапазоном Next.js через context7 при каркасе; фолбэк Node 22 LTS).
- **Пакетный менеджер:** pnpm; lock-файл коммитим.

## 6. Модель данных (Prisma + Postgres)

- `User { id, email, passwordHash, role: admin|viewer, createdAt, updatedAt }`
- `Server { id, name, tag?, baseUrl, apiSkEnc, insecureTLS: bool, createdAt, updatedAt }` — `apiSkEnc` хранится **только зашифрованным**.
- `ServerStatus { serverId (1:1), online: bool, cpu?, mem?, disk?, projectCount?, lastCheckedAt, error? }` — кеш.
- `AuditLog { id, userId, serverId?, action, target?, result, createdAt }`.
- Таблицы Auth.js (User/Account/Session/VerificationToken) — при использовании БД-адаптера.

## 7. Безопасность (критично, см. PROJECT_RULES §7)

- **Шифрование `api_sk` в покое:** AES-256-GCM, ключ из `APP_ENCRYPTION_KEY` (env, 32 байта). Расшифровка только на сервере (Server Actions/воркер).
- **Auth.js v5** + проверка роли на **каждом** серверном действии (`viewer` = read-only; мутации — только `admin`).
- **Валидация (zod)** всех входных данных до похода в панель (защита от SSRF/инъекций путей/портов).
- Ни один секрет не попадает в клиентский бандл / RSC-payload.
- Поддержка самоподписанных сертификатов панелей (`insecureTLS` per server), как в существующем клиенте.
- Аудит всех мутаций (`AuditLog`).

## 8. Слой прокси/API

- Базируется на `examples/javascript/aapanel-client.ts` (режим `api_sk`, подпись `request_token=md5(request_time+md5(api_sk))`).
- Серверный модуль `lib/aapanel`: создаёт клиента **под каждый Server** (расшифровав креды) и даёт типизированные методы по разделам (проекты, БД, файлы, ftp, cron, firewall, система).
- Server Actions (`server/actions/*`) оборачивают вызовы: auth+роль → zod-валидация → вызов клиента → нормализация ошибок → `AuditLog`.
- Типы ответов панели — по снятой документации; для не снятых типов проектов (Proxy/Python/Go) API добираем рецептом при реализации их поддержки.

## 9. Навигация и маршруты (App Router)

- `/login` — вход (Auth.js).
- `/servers` — Уровень 1: плотная таблица серверов (поиск, фильтр тег/статус, сортировка, пагинация, живые обновления по SSE; admin: «＋ сервер»).
- `/servers/[id]` — Уровень 2, layout с левой навигацией разделов; вкладка по умолчанию — **Обзор** (живые метрики).
- `/servers/[id]/projects` — таблица проектов всех типов (колонка «Тип» + фильтр; построчные действия).
- `/servers/[id]/{databases,files,ftp,cron,firewall}` — итеративно (в MVP — каркас).
- `/settings/users` — admin: управление пользователями/ролями.
- `/api/sse/...` — SSE route handler(ы) для живых обновлений.

## 10. Компоненты UI

- **AppShell** (верхняя панель + хлебные крошки), **DataTable** (переиспользуемая плотная таблица на TanStack Table: липкая шапка, компактные строки, построчные действия, виртуализация), **StatusBadge**, **MetricBar** (CPU/RAM/диск), **ConfirmDialog**, **Toaster**, формы (react-hook-form + zod).
- **Анимации** (Framer Motion): появление/удаление строк, переходы вкладок, смена статуса — тонко, без потери производительности.

## 11. Обработка ошибок и логирование

- Единый нормализованный тип ошибки из слоя прокси: `network | timeout | auth | panel_error` со `status`+сообщением.
- Понятные тосты + inline-состояния; `error.tsx` границы по маршрутам; оффлайн-серверы показываются аккуратно.
- Серверное структурное логирование (pino); аудит мутаций.
- Ретраи с backoff для временных ошибок панели в воркере.

## 12. Структура репозитория

Моно-репо: доки остаются в корне, приложение — в `web/`.

```
docs/ … (существующая документация API, остаётся)
examples/javascript/aapanel-client.ts (база клиента)
web/
  app/            # App Router: login, servers, servers/[id]/{overview,projects,…}, settings/users, api/sse
  lib/            # aapanel (client+методы), crypto (AES-GCM), auth (Auth.js+guards), db (Prisma), validation (zod), i18n
  server/actions/ # Server Actions (по разделам) с auth+аудитом
  components/      # ui (shadcn), DataTable, AppShell, …
  worker/          # точка входа воркера (node-cron)
  prisma/schema.prisma
  messages/{ru,en}.json
  Dockerfile  docker-compose.yml  package.json  …
```

## 13. Деплой (двойной режим)

- **Без Docker:** `pnpm install`; dev — `pnpm dev` (Windows); прод (Ubuntu) — `pnpm build && pnpm start` + процесс воркера (`pnpm worker`) под pm2/systemd; Postgres локально или внешний; миграции `prisma migrate deploy`.
- **С Docker:** `docker compose up` — сервисы `app` + `worker` + `postgres` (общий образ, разные команды; Next.js standalone output).
- Env: `APP_ENCRYPTION_KEY`, `AUTH_SECRET`, `DATABASE_URL` и т.д. в `.env` (в `.gitignore`); `.env.example` с плейсхолдерами.
- HTTPS — внешний реверс-прокси (nginx/caddy), вне MVP-кода.

## 14. Тестирование

- **Unit (Vitest):** crypto (шифрование), валидация (zod), маппинг клиента, проверки ролей.
- **Интеграция:** Server Actions с замоканным клиентом панели; Prisma на тестовой БД.
- **E2E (Playwright):** логин → таблица серверов → заход в сервер → действие над проектом.
- Lint/format/typecheck; «готово» только при зелёных тестах (PROJECT_RULES §13).

## 15. Порядок сборки MVP (фазы)

1. **Фундамент:** `web/` каркас (Next.js+TS, Tailwind+shadcn), Prisma+Postgres, Auth.js (admin/viewer), Docker + bare-metal, каркас i18n (RU/EN), базовый AppShell.
2. **Серверы (admin): ✅ ГОТОВО (2026-06-09).** CRUD сервера (шифр. креды) + таблица серверов из кеша (TanStack v8: сортировка/ширина/скрытие колонок, server-side пагинация/поиск/фильтр через URL) + ручное обновление (одной строки и видимой страницы — гибрид «живая видимая страница»). Роли: viewer — только чтение, мутации только admin. План: `docs/superpowers/plans/2026-06-09-phase-2-servers.md`.
3. **Воркер + кеш + SSE:** фоновый опрос с ограниченной параллельностью, живой дашборд.
4. **Обзор сервера** (живые метрики) + **Проекты** (все типы; чтение + старт/стоп/рестарт/логи для Node; остальные типы — по мере снятия API).
5. **Итерации:** Базы данных, Файлы, FTP, Cron, Firewall; UI аудита; полировка/анимации.

## 16. Известные пробелы и риски

- **API не снят** для типов проектов Proxy/Python/Go и для записи firewall/WAF/SSH/SSL — добирать рецептом при реализации соответствующих фич (под явной санкцией для security-операций).
- **`api_sk` против `/v2/...`** проверялся для Node; для прочих модулей при реализации подтвердить ключом (сейчас часть снята на сессии).
- Масштаб >1000 серверов потребует выноса опроса в очередь/воркеры — граница заложена, реализация при необходимости.
- Точная версия Node/совместимость с Next.js — закрепить через context7 при старте каркаса.
