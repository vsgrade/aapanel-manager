# Аутентификация

[English version](../en/authentication.md) · [⌂ Главная](../../README.ru.md)

Методы управления Node.js-проектами (`/v2/project/nodejs/...`) работают через **сессионный токен** панели. Это тот же способ, которым пользуется веб-интерфейс в браузере.

---

## Базовый URL

```
https://<СЕРВЕР>:<ПОРТ>/<SESSION_TOKEN>/v2/project/nodejs/<метод>
```

| Часть | Что это |
|-------|---------|
| `<СЕРВЕР>` | IP или домен сервера с aaPanel |
| `<ПОРТ>` | порт панели (например `41192`) |
| `<SESSION_TOKEN>` | сессионный токен `apsess_...` (см. ниже) |

**Метод HTTP:** `POST` для всех запросов.

---

## Где взять сессионный токен

1. Залогинься в aaPanel в браузере.
2. Посмотри на **адресную строку**:
   ```
   https://192.168.0.10:41192/apsess_xxxxxxxxEXAMPLExxxxxxxx/...
                              └──────────── это и есть токен ───────────┘
   ```
3. Кусок `apsess_...` — твой `SESSION_TOKEN`.

> ⚠️ **Токен временный.** Он меняется при каждом новом входе в панель и со временем истекает. Поэтому **нельзя зашивать его в код намертво** — для постоянной автоматизации приложение должно либо логиниться программно и обновлять токен, либо использовать официальный `api_sk` (см. [overview.md](overview.md)).

## Формат тела запроса

Все параметры передаются как **URL-encoded JSON** в поле `data`:

```
Content-Type: application/x-www-form-urlencoded

data=<URL-encoded JSON>
```

Например, JSON `{"p":1,"limit":10}` в теле выглядит так:

```
data=%7B%22p%22%3A1%2C%22limit%22%3A10%7D
```

В `curl` это удобно делать через `--data-urlencode`:

```bash
curl -k -X POST "https://<СЕРВЕР>:<ПОРТ>/<SESSION_TOKEN>/v2/project/nodejs/get_project_list" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  --data-urlencode 'data={"p":1,"limit":10,"search":"","re_order":""}'
```

## SSL: самоподписанный сертификат

aaPanel обычно использует самоподписанный сертификат, поэтому стандартная проверка SSL не пройдёт. Для тестов:

- `curl` — флаг `-k`;
- Node.js / fetch — отключить проверку сертификата для этого хоста.

> ⚠️ Отключение проверки SSL приемлемо для локальных тестов. В production лучше настроить доверенный сертификат панели или явно добавить её CA, а не глобально отключать проверку.

---

## Безопасность

- Сессионный токен и тем более `api_sk` — это **секреты**. Не коммить их в git, держи в `.env` (см. [`.env.example`](../../.env.example)).
- Любые запросы к панели должны идти **со стороны сервера**, а не из браузера пользователя: иначе секреты утекут в клиентский код, плюс мешают CORS и whitelist по IP.
