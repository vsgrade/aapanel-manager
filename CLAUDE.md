## Главное правило — ПРОПОРЦИОНАЛЬНОСТЬ
Глубина анализа и необходимость согласования зависят от риска изменения. Не превращай тривиальную правку в простыню анализа — это обесценивает правила. Но на 🔴 без подтверждения не приступай.

| Уровень | Что это | Что делаешь |
|---------|---------|-------------|
| 🟢 Тривиальное | опечатка, правка текста доки, формулировка, пример | делай сразу, кратко отметь что сделал |
| 🟡 Среднее | новый раздел доки, новый метод/функция, небольшой рефактор | короткий план (затронутые файлы + риски) → делай |
| 🔴 Высокое | архитектура, зависимости/пакеты, публичный API, удаление кода или функций, структура проекта, бизнес-логика, работа с секретами | **СТОП** → полный анализ → **жди подтверждения** |

## Формат перед 🔴-изменением
1. Анализ задачи 2. Затронутые файлы 3. Риски / что может сломаться 4. Альтернативы 5. Рекомендуемый вариант и почему 6. План 7. Ожидаемый результат → жди «ок».

## Production-ready only
Без костылей, заглушек, mock в проде, хардкода, magic numbers, `any` без причины, проглоченных ошибок, `console.log` в проде, утечек секретов. Делаешь временно — **явно помечай и согласуй**.

## Безопасность секретов (критично для aaPanel)
Никогда не коммить `api_sk`, пароли, токены, IP/URL реальных панелей. Только `.env` (+ `.env.example`), всегда в `.gitignore`. В документации — плейсхолдеры (`<API_KEY>`, `<СЕРВЕР>`, `<SESSION_TOKEN>`), не реальные значения.

## Документирование API aaPanel — ПОЛНОТА
Открыл раздел панели — снимай **всё**, что в нём есть: список, создание, удаление, перезапуск, переименование, права, бэкап, импорт/экспорт, настройки и любые другие действия. По **каждому подтипу отдельно** — у подтипов разные API (подтверждено: MySQL `/v2/data?action=getData` ≠ PostgreSQL `/v2/database/pgsql/get_list`). Ничего не пропускай. Детали — [PROJECT_RULES.md](PROJECT_RULES.md) §15.
